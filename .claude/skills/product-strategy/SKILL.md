## 1. 타겟 페르소나

### 주 페르소나: 김사장 (음식점/카페 자영업자)

```
이름: 김사장 (대표 페르소나)
나이: 35~55세
업종: 한식 음식점 또는 카페
매출: 월 2,000만~8,000만원
과세유형: 일반과세자
세무 지식: 낮음~중간 (세무사에게 위임하거나 직접 홈택스 신고)
디지털 리터러시: 중간 (카카오톡, 배달앱은 능숙)
```

**핵심 페인포인트:**

| 순위 | 페인포인트 | 시기 | 강도 |
|------|-----------|------|------|
| 1 | **세금폭탄 공포** — 납부 시기에 예상보다 큰 세금 고지 | 1월, 7월, 5월 | 극심 |
| 2 | **일상적 불안** — 지금 장사는 잘 되는데 세금이 얼마나 나올지 모름 | 연중 | 중간 |
| 3 | **공제 누락** — 의제매입세액공제 등 받을 수 있는 공제를 모르고 놓침 | 신고 시 | 높음 |
| 4 | **현금 흐름** — 세금 납부를 위한 자금 준비 시점을 모름 | 납부 전 | 높음 |
| 5 | **복잡한 홈택스** — 직접 신고하고 싶지만 홈택스가 어려움 | 신고 시 | 중간 |

**행동 패턴:**
- 매출은 카드사 앱, POS기, 배달앱에서 확인
- 세무사에게 월 11~22만원 기장료 지불 (또는 직접 신고)
- 세금 관련 정보는 네이버 검색, 유튜브, 세무사 블로그에서 습득
- 앱 설치 의사결정은 "무료", "간단", "바로 결과" 키워드에 반응

### 부 페르소나: 이실장 (매장 관리자)

```
나이: 25~40세
역할: 사장 대신 매출/지출 관리
니즈: 사장에게 세금 예상액을 보고해야 함
특징: 디지털 리터러시 높음, 빠른 입력 선호
```

## 2. 경쟁 포지셔닝

### 경쟁 지도

```
전문 서비스 (비쌈)
    │
    │  자비스 (월8만, 기장대행)
    │  비즈넵케어 (수수료, AI+세무사)
    │
세금 신고 ─────────────────────────── 세금 예측
    │                                    │
    │  삼쩜삼 (환급%, 대중적)             │
    │  SSEM (3.3만, 정밀 신고)           │
    │  캐시노트 (매출관리+간단VAT)        │
    │  손택스 (무료, 공식, 복잡)          │
    │                                    │  ★ 세금레이더
    │                                    │  (무료, 오프라인, 음식점특화)
셀프 서비스 (무료/저가)
```

### 세금레이더만의 포지션

**"신고 전에 미리 아는 세금" — 연중 상시 세금 예측 도구**

| 차별점 | 세금레이더 | 삼쩜삼/SSEM | 캐시노트 |
|--------|-----------|------------|---------|
| 목적 | 예측·계획 | 신고·환급 | 매출관리 |
| 시점 | 연중 상시 | 신고 시즌 | 연중 |
| 비용 | 무료 | 신고 시 유료 | 부분 유료 |
| 오프라인 | 100% | 불가 | 불가 |
| 개인정보 | 기기에만 저장 | 홈택스 인증 필요 | 계정 필요 |
| 음식점 특화 | 의제매입 1등 기능 | 부가 기능 | 없음 |
| 시뮬레이션 | what-if 시뮬레이터 | 없음 | 없음 |

### 경쟁이 아닌 관계

세금레이더는 신고 도구가 **아니다**. 삼쩜삼/SSEM과 보완 관계:
- 세금레이더로 연중 계획 → 신고 시즌에 SSEM/삼쩜삼으로 신고
- 또는 세금레이더로 예상 확인 → 세무사에게 위임

**절대 하지 말 것:** 세금 신고 기능 추가. 이는 경쟁 우위를 버리고 레드오션에 진입하는 것.

## 3. 기능 우선순위 프레임워크

### RICE 스코어 기준

| 요소 | 기준 | 점수 범위 |
|------|------|----------|
| **R**each (도달) | 이 기능을 사용할 사용자 비율 | 1~10 |
| **I**mpact (영향) | 사용자 만족도/리텐션 기여 | 1~5 |
| **C**onfidence (확신) | 구현 결과에 대한 확신도 | 0.5~1.0 |
| **E**ffort (노력) | 개발 소요 (낮을수록 좋음) | 1~10 |

**RICE = (R × I × C) / E**

### 기능 평가 예시

| 기능 | R | I | C | E | RICE | 판정 |
|------|---|---|---|---|------|------|
| 로컬 알림 (납부 D-7) | 10 | 4 | 0.9 | 3 | 12.0 | **v1.1 필수** |
| 계산 결과 공유/내보내기 | 7 | 3 | 0.8 | 3 | 5.6 | v1.1 권장 |
| 간이과세자 전용 계산 | 4 | 5 | 0.7 | 5 | 2.8 | v1.2 |
| 홈택스 화면 안내 딥링크 | 8 | 2 | 0.9 | 2 | 7.2 | **v1.1 권장** |
| 다중 사업장 | 2 | 4 | 0.6 | 8 | 0.6 | v2.0+ |
| 배달앱 매출 자동 연동 | 6 | 4 | 0.3 | 9 | 0.8 | 보류 |
| 세금 신고 기능 | 10 | 5 | 0.3 | 10 | 1.5 | **안 함** |

### 기능 제안 시 체크리스트

새 기능 아이디어가 나오면 다음을 먼저 확인:

```
□ 이 기능은 "예측/계획" 영역인가? (신고/환급이면 거절)
□ 음식점/카페 사장에게 실질적 가치가 있는가?
□ 오프라인에서 작동 가능한가? (서버 필요하면 신중 검토)
□ 기존 UX 복잡도를 높이지 않는가?
□ 1주일 이내 구현 가능한가? (아니면 범위 축소)
```

## 4. 유저 저니 맵

### 핵심 여정 7개

```
① 첫 사용 온보딩:  Splash → 가치제안 → 사업장정보 → 첫매출 → 첫결과
② 매출 입력:       레이더/데이터탭 → 매출입력 → 저장 → 복귀
③ 경비 입력:       레이더/데이터탭 → 경비입력 → 저장 → 복귀
④ 의제매입 입력:   레이더/데이터탭 → 의제매입입력 → 저장 → 복귀
⑤ 세금 확인:       레이더 → 세금카드 탭 → 상세분석 화면
⑥ 시뮬레이션:      레이더 → 시뮬레이터 → 슬라이더 조작
⑦ 정밀 종소세:     레이더/데이터탭 → 5단계 위저드 → 결과
```

### "아하 모먼트"

> **첫 결과 화면에서 매출만 입력했는데 부가세 예상액이 나오는 순간.**

이 순간이 앱의 가치를 전달하는 핵심 지점. 최적화 방향:
- 예상 범위가 너무 넓으면 ("100만~800만") 아하 모먼트가 약해짐 → 범위 좁히는 유도 메시지 강화
- 구체적 금액 + "이만큼 준비해두세요" 행동 제안이 효과적

### 리텐션 앵커

| 앵커 | 현재 구현 | 강도 | 개선 방향 |
|------|----------|------|----------|
| 납부 기한 배너 | SeasonBanner (30일 이내) | **강** | 로컬 알림 추가 |
| 정확도 미완성 | AccuracyGauge | 중 | "매출 입력하면 정확도 +20%" 넛지 |
| 절세 팁 변화 | 동적 팁 텍스트 | 중 | "바로 입력하기" CTA 추가 |
| 월초 매출 정산 | 없음 | 약 | 월초 알림 "지난달 매출 입력하세요" |

### 이탈 위험 지점

| 지점 | 리스크 | 원인 | 대응 |
|------|--------|------|------|
| 온보딩 첫 매출 | 중간 | 프리셋에 본인 매출 없음 | 직접 입력 필드 추가 |
| 레이더 첫 화면 | 중간 | 정보 과부하 | 빈 상태 시 단순한 CTA만 |
| 의제매입 입력 | 높음 | 용어 자체를 모름 | 진입 시 간단 설명 모달 |
| 월간 미복귀 | 높음 | 알림 없음 | 로컬 알림 구현 |

## 5. 버전별 로드맵 기준

### 원칙

> **v1.0은 "쓸 수 있는 앱", v1.x는 "좋은 앱", v2.0은 "필수 앱"**

### v1.0 (현재) — MVP 출시

핵심 가치만 전달. "매출 넣으면 세금 예측" 루프가 작동하면 충분.

포함: 부가세/종소세 예측, 매출·경비·의제매입 입력, 상세 분석, 시뮬레이터, 정밀 종소세, 용어사전, 정확도 게이지
제외: 알림, 공유, 다중 사업장, 신고 기능

### v1.1 — 리텐션 강화 (출시 후 2~4주)

목표: 재방문율 향상. 사용자가 매월 돌아오게 만든다.

| 기능 | RICE | 근거 |
|------|------|------|
| 로컬 알림 (납부 D-7, 월초 입력) | 12.0 | 리텐션 최대 레버 |
| 홈택스 안내 딥링크 | 7.2 | 실용성 높음 |
| 계산 결과 스크린샷/공유 | 5.6 | 세무사 상담 시 활용 |
| 월별 데이터 삭제 기능 | — | P2-2 품질 |
| 시뮬레이터 빈 상태 안내 | — | P2-3 품질 |

### v1.2 — 정확도 향상 (출시 후 1~2개월)

목표: 계산 정확도 향상. 사용자의 데이터가 풍부해질수록 가치가 커진다.

| 기능 | 근거 |
|------|------|
| 부양가족 인원수 세분화 (bool→int) | 종소세 정확도 |
| 간이과세자 전용 계산 | 신규 사용자 확장 |
| 금융소득종합과세 대응 | 정밀 엔진 완성 |
| 납부 실적 기록 | 다음 반기 예측 개선 |

### v2.0 — 확장 (출시 후 3~6개월)

목표: "세금 레이더" = 자영업자 세금 관리의 대명사.

| 기능 | 근거 |
|------|------|
| 다중 사업장 지원 | 부부 공동 운영, 2호점 |
| 연도별 데이터 비교 | 추세 파악 |
| 세금 리포트 PDF 내보내기 | 세무사 상담용 |
| 홈 화면 위젯 (D-day) | 일상 접점 |

## 6. ASO (앱 스토어 최적화) 가이드

### 앱 이름 + 부제

```
앱 이름: 세금레이더
부제목: 음식점·카페 부가세·종소세 예측
```

### 핵심 키워드 (검색 우선순위)

| 키워드 | 검색 의도 | 경쟁도 |
|--------|----------|--------|
| 부가세 계산기 | 즉각적 니즈 | 높음 |
| 음식점 부가세 | 업종 특화 | 중간 |
| 카페 세금 | 업종 특화 | 낮음 |
| 의제매입세액공제 | 니치 | 매우 낮음 |
| 자영업자 세금 | 넓은 타겟 | 높음 |
| 부가세 예측 | 계획 의도 | 매우 낮음 |
| 세금 미리 계산 | 계획 의도 | 매우 낮음 |

**iOS 키워드 필드 (100자):**
`부가세,종소세,자영업자,음식점세금,카페세금,의제매입,세금계산기,절세,사장님`

### 스크린샷 전략

| 순서 | 내용 | 메시지 |
|------|------|--------|
| 1 | 레이더 대시보드 (부가세 금액 강조) | "다음 부가세, 미리 확인하세요" |
| 2 | 의제매입 공제 절감 효과 | "면세 식재료로 부가세 절감" |
| 3 | 정확도 게이지 + 데이터 입력 | "입력할수록 정확해져요" |
| 4 | 세금 캘린더 + D-day | "납부 기한, 놓치지 마세요" |
| 5 | 시뮬레이터 | "매출이 늘면 세금은?" |

### 앱 설명 구조

```
첫 줄: 페인포인트 공감
  "부가세 신고일이 다가올 때마다 얼마나 나올지 몰라 불안하셨나요?"

둘째 줄: 해결책 한 문장
  "세금레이더는 음식점·카페 사장님을 위한 세금 예측 앱입니다."

핵심 기능 (불릿):
  • 매출만 입력하면 부가세·종소세 예상액 즉시 확인
  • 의제매입세액공제 자동 계산 — 면세 식재료로 절세
  • 시뮬레이터로 "매출이 늘면 세금은?" 미리 확인
  • 세금 캘린더로 납부 기한 관리

신뢰 요소:
  • 100% 오프라인 — 인터넷 없이도 작동
  • 개인정보 외부 전송 없음 — 기기에만 저장
  • 완전 무료 — 숨겨진 비용 없음

CTA:
  "지금 설치하고 다음 부가세 예상액을 확인하세요!"
```

### 리뷰 유도 타이밍

- 첫 번째 VAT 반기 계산 완료 후 (높은 만족 순간)
- 정확도 70% 이상 달성 시
- **하지 않을 것:** 첫 실행, 온보딩 중, 데이터 입력 중

## 8. 절대 하지 말 것 (Anti-Patterns)

| Anti-Pattern | 이유 |
|---|---|
| 세금 신고 기능 추가 | 삼쩜삼/SSEM과 정면 경쟁. 규제 리스크. 핵심 포지션 훼손 |
| 서버 구축 | 오프라인 = 핵심 차별점. 서버 추가 시 개인정보 동의, 비용 발생 |
| 홈택스 인증 연동 | 사용자 신뢰("내 데이터 안 빼감") 훼손 |
| 너무 많은 업종 지원 | 음식점/카페 특화 = 경쟁력. 범용화하면 차별점 소멸 |
| 유료 구독 (v1.x) | 초기 사용자 확보가 우선. 과금은 충분한 사용자 기반 후 |
| 복잡한 입력 필드 추가 | 입력 부담 ↑ = 이탈 ↑. 간단함이 핵심 UX |
